## Context

Brilliant Sort already compiles a deterministic headless C++20 game core to WebAssembly, but it has no runtime music or effects. The approved flagship direction is one large Tux mosaic in a wordless pixel cavern. The requested audio identity is original 16-bit-era music and tactile effects generated by C++ oscillators, with a hand-authored but constrained algorithmic composition rather than imported recordings or unconstrained random melody.

Browser audio is not canonical gameplay state: device sample rate, output latency, visibility suspension, and autoplay permission vary. Emscripten's Wasm Audio Worklets API can run C/C++ processing in a dedicated realtime-priority worklet with 128-frame callbacks, while browsers require an audible `AudioContext` to resume from a user interaction. This subsystem therefore needs its own WASM target, lifecycle, and verification boundary.

The sibling `rebuild-tux-mosaic-stage` change remains authoritative for core transitions and visual state. The shared boundary is a presentation-derived `AudioCue` stream carrying accepted transition facts; no audio clock, voice, score position, or mute preference enters `GameState`, `LevelSpec`, canonical dumps, or replay decisions.

## Goals / Non-Goals

**Goals:**

- Implement an allocation-free C++ synth with a native offline renderer and separate Emscripten Wasm AudioWorklet runtime.
- Produce an original 16-bit-era Tux theme from oscillator/wavetable synthesis with no runtime sample assets.
- Combine hand-authored harmony/motifs with deterministic constrained variation, fixed score seed, and progress-driven arrangement layers.
- Synthesize consistent selection, storage, placement, rejection, compaction, and victory cues from presentation facts.
- Start audio on the first real puzzle interaction without a start screen and provide an accessible in-world mute crystal.
- Prove fixed-rate PCM determinism offline and verify realtime safety/browser lifecycle independently from gameplay parity.
- Fail silent: audio initialization or rendering failure must never block or alter the game.

**Non-Goals:**

- Coupling audio timing to `BrilliantSortCore`, adding audio fields to canonical state, or making sound a gameplay input.
- Streaming files, MP3/OGG/WAV assets, third-party synth libraries, browser `OscillatorNode` composition, MIDI input, or user music editing.
- Unbounded procedural composition, network generation, machine-learning audio, dynamic downloads, or a playlist for multiple levels.
- Spatial 3D audio, voice chat, microphone access, or a general DAW/tracker UI.

## Decisions

### 1. Build a separate C++ audio module

`PixelAudioEngine` is a pure C++ library with `Initialize`, `LoadScore`, `PushCue`, `SetMuted`, and `Render` boundaries. A native executable renders fixed-duration signed 16-bit PCM for inspection/tests. A distinct Emscripten target produces `brilliant-sort-audio.wasm` and initializes one Wasm AudioWorklet processor; the existing `brilliant-sort-core.wasm` and C ABI remain unchanged.

The browser owns lifecycle only: load module, request/resume context from a gesture, map transitions to cues, and expose mute. Audio failure resolves to a disabled/silent port rather than a second gameplay path.

Alternative considered: compile synthesis into `BrilliantSortCore`. Rejected because realtime clocks and browser lifecycle would pollute the deterministic rule boundary and force game tests to initialize audio.

### 2. Use fixed-point musical state and 16-bit reference output

Voices use 32-bit phase accumulators, fixed-point envelopes, integer tick/sample scheduling, and a widened integer mix bus before saturation to signed 16-bit PCM. Realtime Web Audio output converts the final sample to `[-1, 1]` float at the boundary. Musical event order is sample-accurate at the actual context rate; exact PCM hashes are normative only for the fixed 48kHz offline profile.

The engine preallocates a fixed voice pool, score storage, cue queue, and working buffers before rendering. `Render` performs no allocation, lock, I/O, logging, parsing, or blocking call.

Alternative considered: use `std::sin`, floating envelopes, and dynamic voice containers. Rejected because cross-platform rounding and allocation behavior make deterministic offline verification and realtime safety weaker.

### 3. Generate timbre from a bounded oscillator family

The initial engine provides pulse waves with selectable duty, triangle, multi-band saw/wavetable, sine-table two-operator FM, and a fixed-tap LFSR noise source. ADSR, vibrato, portamento, duty sweep, arpeggio, and simple gain staging are fixed-capacity instrument parameters. Band-limited/multi-band tables prevent harsh high-note aliasing while preserving a 16-bit-era character.

Music and effects share the engine/transport but reserve separate voice budgets so a multi-gem chord cannot steal the bass clock or stall the score. A hard limiter/saturator protects output; authored gains remain below the limiter in the accepted offline render.

Alternative considered: use only square waves. Rejected because it produces a narrow 8-bit-beeper identity rather than the requested richer 16-bit sound.

### 4. Combine authored motifs with deterministic constrained variation

`tux-01.music.json` stores tempo, ticks per beat, scale, chord progression, motif note offsets, rhythm masks, instrument assignments, arrangement sections, loop boundaries, progress thresholds, and one fixed seed. The C++ sequencer may apply only declared transformations such as motif rotation, octave displacement, chord-tone substitution, bounded note omission, and arpeggio order. It cannot choose notes outside the configured scale/chord grammar or mutate tempo/structure arbitrarily.

The fixed seed, loop index, progress layer, and score data fully determine every variation. The accepted 90–120 second super-loop therefore repeats deterministically while sounding less mechanical than a single copied pattern. Human review approves the rendered melody/harmony; passing algorithmic constraints alone is insufficient.

Alternative considered: fully handwritten note events only. Rejected because the user explicitly accepts constrained algorithmic music and one flagship level benefits from controlled variation. Unconstrained random melody is also rejected because it is not musically reviewable or replayable.

### 5. Keep the cue bridge presentation-only and ordered

The browser maps post-dispatch facts to fixed-size, monotonically sequenced `AudioCue` packets:

- `selection(color, count)`
- `shelf_store(color, count)`
- `target_place(color, count)`
- `shelf_compact(movedCount)`
- `rejected(code)`
- `progress(matched, total)`
- `won`

Cues enter a bounded single-producer/single-consumer queue through Emscripten worklet messaging. Overflow drops/coalesces low-priority selection/progress cues but never blocks the main thread; `won` and mute controls have reserved delivery. The audio engine may change sound/arrangement only and cannot dispatch a game command.

The visual proposal may change animation timing without changing cue semantics: cues derive from accepted core results, not animation callbacks. This lets both OpenSpecs implement independently against the same boundary.

### 6. Use one progress-layered Tux score and synthesized effects

The score exposes arrangement layers at configured repair thresholds, initially targeted near 25%, 50%, and 75% matched progress. Crossing a threshold schedules the next layer at a musical boundary instead of switching mid-sample. `won` prevents another loop section from starting, lets the current safe boundary resolve, and schedules one original victory fanfare.

Selection pitch is stable by gem color; Shelf storage uses a lower seated articulation; target placement forms bounded intervals/chords from count; compaction emits quantized ticks; rejection uses a short filtered noise chirp. Cue density and gain are capped so large connected components do not generate dozens of simultaneous voices.

### 7. Resume on first puzzle interaction and use a cavern crystal for mute

Audio module loading may begin with the App, but the context resumes only inside the first pointer or keyboard activation of an actionable board/Shelf cell. That same gesture still dispatches its gameplay command; there is no start overlay. If resume is rejected or worklet creation fails, the game continues silently and exposes failure only through diagnostics/accessible status.

One decorative cave crystal becomes a real semantic button layered into the scene with at least the project touch target, `aria-pressed`, dynamic accessible name, keyboard activation, and visible lit/dim states. Muting ramps master gain briefly to avoid clicks and persists locally; unmuting from a suspended context is itself a user gesture and resumes safely.

Alternative considered: omit an in-app mute control and rely on system volume. Rejected because users need product-level control over continuous music. A dashboard speaker icon is rejected because it conflicts with the approved wordless cavern.

### 8. Handle suspension without pretending it is gameplay time

When the document/audio context suspends, rendering stops according to browser policy. On resume, the engine continues at a deterministic musical boundary or restarts the current pattern according to one configured policy; it does not simulate missed wall-clock samples or advance gameplay. Visibility changes and device-rate changes cannot emit gameplay commands.

## Risks / Trade-offs

- [Wasm AudioWorklet support/build differs from the existing main-thread module] → Use a separate Emscripten target, prove a minimal worklet boot before score work, and keep a silent port fallback.
- [Autoplay blocks sound on load] → Resume only from the first actual puzzle gesture and never require sound for play.
- [Realtime queue overflow under large moves] → Aggregate counts, reserve critical cue capacity, coalesce progress, and drop only low-priority feedback without blocking.
- [Naive oscillators alias or fatigue listeners] → Use multi-band wavetables, conservative register ranges, gain budgets, and human headphone/speaker review.
- [Algorithmic variation produces weak music despite deterministic tests] → Constrain transformations to authored harmony/motifs and make rendered-listening approval an explicit gate.
- [Floating/browser differences break exact hashes] → Hash only fixed-rate native 16-bit reference PCM; browser tests assert lifecycle, cue behavior, bounds, and audible node processing rather than cross-device waveform equality.
- [Continuous music conflicts with the no-HUD direction] → Use a real in-world crystal control with accessible semantics, not a floating control panel.
- [Audio work delays the Tux visual change] → Keep modules and acceptance independent, share only `AudioCue`, and allow both changes to proceed in parallel.

## Migration Plan

1. Add the native synth/offline renderer and fixed 48kHz golden render before browser integration.
2. Add the separate Emscripten AudioWorklet target and prove silent oscillator startup/resume/failure fallback.
3. Define the shared `AudioCue` bridge and synthesized effects without adaptive music.
4. Add the constrained score compiler/data, base loop, progress layers, and victory fanfare.
5. Convert a cavern crystal into the accessible persisted mute control.
6. Verify native determinism, zero-allocation rendering, browser lifecycle, supported engines, background/resume, and production hosting.
7. Roll back by disabling/removing the audio module import; gameplay WASM, fixture, and canonical state require no migration.

## Open Questions

No product decisions remain open before implementation. Exact melody, harmony, tempo within the proposed range, instrument patches, cue pitches, mix balance, and progress thresholds are audio-review details delegated to implementation and human listening approval.
